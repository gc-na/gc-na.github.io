반도체 제조공정은 완벽하지 않기에 항상 결함이 있는 Chip이 있고, 불량과 양품을 거를 수 있는 기준이 필요합니다. ATPG, Automatic Test Pattern Generation. 한국어로 하면, 자동 테스트 패턴 생성으로 "반도체 입력 값 + 정상 반도체에서 나와야 할 출력 값"을 EDA Tool로 만듭니다.

​

![](https://postfiles.pstatic.net/MjAyMzA4MjBfMjA1/MDAxNjkyNDc0OTQyMDY4.GTKb-LLf6FVCmeMLYlC_s2feh0kQryRrdPT8byvpAKkg.q15TaAg4SWEtQtOMXumu2v5ULI0cEulbWFALgkb5zrMg.PNG.gc_na/image.png?type=w966)

​

결함이 있는 Chip이 의료 기기, 항공 우주, 방산, 자율 주행과 같은 중요한 시스템에 사용될 경우 발생할 수 있는 잠재적 결과를 상상해보면 끔찍합니다. **소비자에게 가기 전에 모든 공정 결함 칩들은 확인이 되어야합니다.**

​

그런데, 최신 반도체 칩은 이 테스트라는게 쉽지 않습니다.

1) 나노 공정라는 단위에서 만들어지고,

2) 수십 수백 수천억 개의 Transistor로 이뤄지고,

3) 1ns (unit : nano)보다도 짧은 주파수로 동작하며,

4) 반도체 칩의 크기가 작고, Chip에서 사용 할 수 있는 Input, Output 개수가 많지 않으며,

5) SoC, SiC 등으로 이뤄진 하나의 Chip에 엄청나게 많은 기능들이 들어있으며, Chip Test는 엄청난 전력 소비를 하기 때문에 Test 과정 중에 Power issue가 있을 수 있고,

6) 이 Chip들을 양산에 성공하려면 짧은 Pattern에 높은 Test coverage를 만들어야 합니다.

​

그렇기 때문에.. 디지털 회로의 테스트의 문제는 "테스트 패턴을 어떻게 효율적으로 생성 할 수 있는지?"가 중요합니다.

​

Pramary Input, output이라고 불리는 Chip의 port만을 갖고 테스트해야해요.

-> 그냥 Chip을 Function vector 넣고 Test해도 되지만, Test coverage와 시간적 비용 때문에 ATPG를 사용합니다.

​

ATPG의 동작과정은,

![](https://postfiles.pstatic.net/MjAyMzA4MjBfNjAg/MDAxNjkyNDcyMDYxMDQ5.cYQkKNjl9f_jPw7SvRvFViYN4M05YvkXfGAjmrU3rSgg.6bTbvg0xM_EzlhZ-KbYg2dR8G2Dz6ww8T9wk3qH_aXIg.PNG.gc_na/image.png?type=w966)

0) Design read : Design과 Test protocol을 읽고,

1) Build Fault model : 회로에서 발생할 수 있는 결함(Fault Model)을 정의합니다.

2) Test Pattern generation: Fault model을 기반으로 manual하게 사람이 만들거나, Tool로 Random generation을 하거나, EDA Tool을 통해 디지털 논리적인 Algorithm을 갖고 테스트 패턴을 생성합니다.

테스트 패턴은, "입력 A에 1 주고.. B에 1 주고.. 이럴때 C가 1이 나와야 정상 Chip" 이런겁니다.

3) 이걸 전체 RTL 혹은 Netlist에 적용합니다.

​

이 논리적인 Algorithm이라는게, 여러가지 있는데.. D-Algorithm만 알아도 기본 컨셉은 다 알 수 있습니다. 아래 동영상 참조하세요! 좀 더 보고싶으시면, LFSR, PODEM을 찾아서 보시면 됩니다.

Stuck At fault (SA):

디지털 회로에서 공정 결함으로 발생할 수 있는 가장 일반적이고 잘 알려진 결함 유형 중 하나입니다. 선 하나가 공정 중에 끊어졌거나, VDD에 붙어버릴 수 있잖아요? 이러면 입력이 뭐든간에 특정 부분이 하나의 값으로 붙어버립니다.

Bridge Fault:

공정 결함으로 하나의 배선이 다른 배선에 전기적으로 연결될 때 발생합니다.

Transition Delay fault (TD):

Stuckat이나 Bridge fault는 SCAN Test에서 동작 하기 때문에, 외부 장비를 연결하여 입력과 출력 패턴을 이용합니다.

외부 I/O를 이용하기에, 고속의 Clock을 PAD를 통해 넣을 수 없기에, 클럭도 저속의 클럭을 사용합니다.

​

근데, 공정 결함으로... 배선이 너무 얇거나, 너무 두껍거나 하는 이유로.. 디지털 논리적으론 문제는 없지만, 물리적인 문제가 있다면,

Wire의 R/C 값이 늘어나거나, Transistor의 문턱전압이 바뀔 수 있습니다. 이러면 Chip 동작 속도에따라 결함이 발견 될 수도 있죠. 이걸 Transition delay fault라고 합니다.

근데 저속의 clock에서는 이런 문제들을 발견 할 수 없겠죠? 그래서 DFT(Design For Test)에서 여러가지 설계 기술을 갖고, Chip test할 때 고속으로 동작 할 수 있도록 설계를 합니다.

​

​

​

Chip 설계는 Function이나 Mission mode라고 불리는.. Chip의 기본적인 동작을 위해 설계됩니다.

그러나 Chip test는 꼭 해야하고, 양산하려면.. automotive 하려면.. 꼭 Test coverage를 높여야 합니다. 근데.. 항상 Function이 우선이다보니, 1) DFT에서 Fault coverage를 만족시키기도 어렵고,

2) Chip test mode의 Timing 조건을 만족 시켜아하고,

3) 이 상태에서도 Power 이슈가 없는지 확인해야하며,

4) Chip test logic은 Serial chain 형태를 이루는데, 이게 실제적인 Cell 배치인 Place & Route 과정을 거치면, Logic design에서는 Serial 했던 5) Chain이 뒤죽박죽 꼬이면서 Timing / PV Issue가 발생하기에, re-ordering 같은 고려도 해줘야합니다.

​

설계/검증에서 Post-Simulation 직전에 시행되는... 후반 단계의 작업입니다.
